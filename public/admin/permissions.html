People and permissions

The S3S admin explains the access model, but GitHub remains the authority that grants and revokes access. This keeps organization credentials out of the CMS.

Access changes happen in GitHub. Only organization owners or repository administrators should use the management links below. Ordinary editors do not need organization-management permission.

Role model

Site editors

Use Decap CMS for routine copy, events, resources, officers, and other approved content. Recommended GitHub role: Write.

Radar maintainers

Maintain the Opportunity Radar feed, source parsers, automation, and protected interface behavior. Recommended GitHub role: Write with code ownership.

Site admins

Manage deployments, repository settings, protections, and technical handoff. Recommended GitHub role: Maintain or Admin.

Organization owners

Grant membership, approve OAuth applications, and preserve ownership continuity. Keep this group small.

Onboarding

  1. Add the person to the appropriate GitHub team.
  2. Give that team the least repository role needed.
  3. Approve RCID S3S CMS for the organization if GitHub reports that OAuth access is restricted.
  4. Have the person sign in at the S3S admin and save a draft change.
  5. For Radar maintainers, review the Radar maintainer guide together.

Offboarding

  1. Remove the person from the GitHub team or repository.
  2. Review open CMS branches and pull requests owned by that person.
  3. Transfer any ongoing Radar maintenance work to another maintainer.
  4. Rotate secrets only if the departing person had access to Cloudflare or OAuth application credentials.
